Create your first project
Organising your testing workflow into projects allows for better organisation of saved recordings and replays
Project workflow allows for easy sharing of testing environments among large teams
Once you have completed input to save your first project, you can start adding new tests
Set up your first test
Tests are linked to a single starting url and can contain many recordings and replays
Tests allow bandwidth and latency throttling, as well as taking screenshots on replay success or failure
Once you have completed input to save your first test, you can start adding new recordings
Start your first recording
Recordings capture and save all user mouse and keyboard actions in the testing browser window until stopped
You can flip between the Record/Replay tab and the test target tab to see your actions being recorded in real time
Unwanted recordings events can be removed via the table interface, with final output saved to local storage for later replay
Replay your first recording
To create your first replay, select a previous recording from menu, add the assertions you want performed, then hit start
The replay reports to the Record/Replay tab, so you can see your tests progressing with identical time intervals as the recording.
If the replay matches your recording, download your replay as Javascript code for inclusion in Jest/Puppeteer testing suites

If you are testing as you are developing, attach id attributes to elements you want to action or to verify. Great care is taken to ensure that recordings can be reliably replayed in any circumstance and with any selectors. You can still bomb-proof your tests and ease your reading of output by adding a unique id to each element you want to test.
When saving your recordings for the first time, choose a description that describes what replays of the recording should be aiming to verify. Replay descriptions are automatically generated from your original recording description. If you have multiple replays and recordings, things will get complicated quickly if you fail to add functional labels to your recordings.
When you start your first recording, you may notice that a popup at the top of the recorded page, alerting you that Record/Replay uses the Chrome Devtools Protocol. If this bothers you, you can paste 'chrome://flags/#silent-debugger-extension-api' into your browser address bar, click enable and then relaunch. The message will not appear again.
When recording is in progress, the mouse Hover action is fired after hovering over an element for more than three seconds. This may be longer than you'd expect, so try counting slowly to three. Or quickly to five. Or something. Don't worry if you collect too many Hover actions, you can delete them from your recording and it will become second nature.
Hover and Text Selection actions are the only recorded actions that allow you to add assertions that can be checked on replay. Text selection operates as you would expect - if you select text on the page during a recording, the element selected is made available for qualified assertions. All other actions are subject to their own verification checks.
Be careful when editing existing tests or recordings. Changing bandwidth and mobile emulation parameters can lead to unpredictable results. Our advice is to create separate mobile and computer recordings for the same test, thereby ensuring that the testing environment remains constant.
