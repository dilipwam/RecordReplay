Organising your testing workflow into projects allows for better organisation of saved recordings and replays
Project workflow allows for easy sharing of testing environments among large teams

Set up your first test
Tests are linked to a single starting url and can contain many recordings and replays
Tests allow bandwidth and latency throttling, as well as taking screenshots and reporting visual changes
Once you have completed input to save your first test, you can start adding new recordings
Start your first recording
Recordings capture and save all user mouse and keyboard actions in the testing browser window until stopped
You can flip between the Record/Replay tab and the test target tab to see your actions being recorded in real time
Unwanted recordings events can be removed via the table interface, with final output saved to local storage for later replay
Replay your first recording
To create your first replay, select a previous recording from menu, add the assertions you want performed, then hit start
The replay reports to the Record/Replay tab, so you can see your tests progressing with identical time intervals as the recording.
If the replay matches your recording, download your replay as Javascript code for inclusion in testing suites

          "Verily the verification of your venture is our vocation, volunteered as votive, so view these verities as virtuous"
          If you are testing as you are developing, attach unique 'id' or 'data-*' attributes to elements you want to action or to verify. As HTML structure can change, you can bomb-proof your tests and ease your reading of output by adding these unique attributes. Don't worry if that is impossible, as redundancy is built into the selector generation and reporting process.
          When saving your recordings for the first time, choose a description that describes what replays of the recording should be aiming to verify. Replay descriptions are automatically generated from your original recording description. If you have multiple replays and recordings, things will get complicated quickly if you fail to add functional labels to your recordings.
          When you start your first recording, you may notice that a popup at the top of the recorded page, alerting you that Record/Replay uses the Chrome Devtools Protocol. If this bothers you, you can paste 'chrome://flags/#silent-debugger-extension-api' into your browser address bar, click enable, then relaunch. The message disappears until Chrome version updates.
          As recording proceeds, the 'Scroll' action fires whenever scrolling has stopped. This may generate more scrolling events than are strictly necessary. If the 'Scroll' action has a final destination, you can reduce the number of events in your recording by deleting any scroll events leading up to the final destination.
          When recording keyboard input, both 'Keyboard' events and 'Input' events are logged. This ensures that recorded actions can be replayed, even when simulated 'Input' change events are disabled, e.g. on login forms. If your replay fires all the relevant input change events, delete the keystrokes from the replay. Otherwise, keep the keystrokes and delete input events.
          The mouse 'Hover' action is fired during recording after hovering over an element for more than three seconds. This may be longer than you'd expect, so try counting slowly to three. Or quickly to five. Or something. Don't worry if you collect too many 'Hover' actions, you can delete them from your recording and it will become second nature.
          'Hover' and 'Text Selection' actions are the only recording actions that allow you to add assertions that can be checked on replay. Text selection is fired by mouse drag NOT double click. Partial text selections within elements cannot be verified so all text content in the target element is recorded. All other actions are subject to their own verification checks.
          If you wish to monitor visual changes on replay, it is important to remember that replay visual regression analysis is sensitive to changes in the size of compared screenshots. The rule is: keep your browser window at a uniform size. Otherwise you will get a lot of false negatives in visual checks. Tests will pass but the visual check image will look messy.
          Be careful when editing existing tests or recordings. Changing bandwidth and mobile emulation parameters can lead to unpredictable results. Our advice is to create separate mobile and computer recordings for the same test, thereby ensuring that the testing environment remains constant.
